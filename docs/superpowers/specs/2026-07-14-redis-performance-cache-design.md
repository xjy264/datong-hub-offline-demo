# Redis 读缓存性能优化设计

## 背景

项目已经部署 Redis 7，并引入 `spring-boot-starter-data-redis`，但业务代码目前没有读写 Redis。当前主要读路径存在两个可优化点：

1. `StationService.listStations()` 每次都重新聚合站点、目录和图片元数据。
2. 聚合站点时会按站点重复查询车间列表；现有 171 个站点会把一次聚合放大为大量重复 SQL。
3. `MapDocumentService` 又实现了一套站点、目录和图片聚合逻辑，地图详情请求重复承担相同成本。

图片二进制已经由浏览器 HTTP 缓存和懒加载处理，继续放入 Redis 会增加内存与序列化成本，不属于本次优化范围。

## 目标

- 让站点聚合、车间列表和地图列表的重复读取优先命中 Redis。
- 把一次站点冷加载的数据库查询压缩为固定数量，消除按站点重复查询车间的 N+1 问题。
- 所有成功写操作完成后立即失效相关缓存，避免用户看到旧数据。
- Redis 暂时不可用时回退到 MySQL，核心页面仍可使用。
- 保持单后端实例的简单架构，不引入分布式锁、消息队列或缓存预热任务。

## 非目标

- 不缓存图片、PDF或其他二进制文件。
- 不把登录态、密码、JWT或权限判断迁移到 Redis。
- 不缓存上传、导入、导出等低频写流程。
- 不缓存完整地图详情，避免跨站点、标记和区间的复杂失效关系。
- 不调整前端页面行为。

## 方案选择

采用 Spring Cache 注解和 `RedisCacheManager`，而不是手写 Redis 键操作。

原因：

- 项目已经包含 Spring Data Redis，无需增加新的 Redis 客户端。
- 读穿缓存、TTL和写后失效可以集中表达，代码量最小。
- 测试可以使用内存 CacheManager 验证命中和失效，不要求单元测试依赖真实 Redis。
- 生产环境仍使用真实 Redis，并通过烟雾测试验证序列化、TTL和故障回退。

## 缓存边界

| 缓存名 | 来源方法 | 键 | TTL | 失效条件 |
| --- | --- | --- | --- | --- |
| `stationViews` | `StationService.listStations()` | 固定键 `all` | 5分钟 | 站点、目录、图片或导入数据发生变化 |
| `workshops` | `WorkshopService.listWorkshops()` | 固定键 `all` | 10分钟 | 车间新增、改名或删除 |
| `mapSummaries` | `MapDocumentService.listMaps()` | 固定键 `all` | 10分钟 | 地图新增、改名或删除 |

所有键使用 `datong-map::` 前缀，防止与同一 Redis 中的其他应用冲突。缓存值使用带类型信息的 JSON 序列化，兼容 Java record、列表和 `LocalDateTime`。

## 读取数据流

### 站点列表

1. 调用 `StationService.listStations()`。
2. `stationViews::all` 命中时直接返回缓存结果。
3. 未命中时，从 MySQL 分别读取车间、站点、目录和图片元数据。
4. 在本次请求内构建车间代码到公开 ID 的映射，不再为每个站点重复查询车间表。
5. 聚合结果写入 Redis，TTL 为5分钟。

冷加载最多执行固定数量的查询；热加载不访问 MySQL。

### 地图详情

`MapDocumentService` 删除内部重复的站点聚合实现，改为调用 `StationService.listStations()`。地图本身、标记和区间仍实时查询 MySQL；体量最大的站点、目录和图片元数据从 `stationViews` 缓存复用。

地图详情不整体缓存，因此站点缓存失效后下一次详情请求即可取得最新站点数据，标记和区间修改也不需要维护额外缓存键。

### 车间与地图列表

车间列表和地图列表按固定键缓存。地图标记或区间变化不会影响地图列表，因此只在地图新增、改名和删除时失效 `mapSummaries`。

## 写入与失效

缓存失效在业务方法成功执行后发生，并使用事务感知的 CacheManager，使失效动作在数据库事务提交后生效。

### 清除 `stationViews`

- 更新站点资料
- 创建站点
- 新建、改名、删除目录
- 上传、删除图片
- 导入旧数据
- 新增、改名、删除车间

### 清除 `workshops`

- 新增车间
- 车间改名
- 删除车间

### 清除 `mapSummaries`

- 新增地图
- 地图改名
- 删除地图

失败或回滚的写操作不得清除或写入缓存。

## Redis 故障处理

Redis 只作为性能增强层，不成为业务可用性的硬依赖：

- 缓存读取失败时执行原方法并查询 MySQL。
- 缓存写入或删除失败时记录告警，但不让业务请求失败。
- Redis 恢复后后续请求自动重新写入缓存。
- Redis 故障期间可能保留的旧缓存由5至10分钟TTL限制最长陈旧时间。
- 后端健康检查不把 Redis 状态作为整体存活条件；MySQL仍是核心依赖。
- Docker Compose 仍启动 Redis，但后端不等待 Redis 健康后才允许启动。

## 配置

- 在应用入口启用 Spring Cache。
- 增加 `spring-boot-starter-cache`，继续复用现有 `spring-boot-starter-data-redis`。
- 新增集中式 Redis CacheManager 配置：键前缀、JSON序列化、TTL和事务感知。
- 增加 CacheErrorHandler，实现 Redis 异常时的 MySQL 回退。
- 关闭 Actuator 的 Redis 整体健康贡献，避免纯缓存故障把后端标记为不可用。
- 暂不修改 Redis AOF、最大内存和淘汰策略；当前缓存规模很小，先避免扩大运维变更范围。

## 测试策略

实施遵循测试先行：

1. 先写缓存命中测试，证明连续两次读取只执行一次底层查询。
2. 先写写后失效测试，证明站点、车间和地图写操作会清除正确缓存。
3. 先写 Redis 异常回退测试，证明缓存故障时原方法仍执行。
4. 更新 `MapDocumentService` 测试，证明地图详情复用 `StationService`，不再维护第二套站点聚合。
5. 运行全部后端单元测试和前端构建。
6. 使用本地真实 Redis 启动后端，验证缓存键、TTL、JSON反序列化和重复请求。
7. 停止 Redis 后再次请求，验证接口继续从 MySQL返回；恢复 Redis 后验证缓存重新建立。

## 验收标准

- Redis 中能够看到带 `datong-map::` 前缀的三个缓存域。
- 第二次调用站点、车间或地图列表时不重复执行对应 SQL。
- 站点冷加载不再按站点数量重复查询车间表。
- 任一相关写操作完成后，下一次读取返回最新数据。
- Redis停止时，地图、站点和车间读取接口仍返回成功响应。
- 现有38个后端测试全部通过，前端生产构建成功。
- 不改变现有API响应结构、图片缓存行为和用户操作流程。

## 发布边界
